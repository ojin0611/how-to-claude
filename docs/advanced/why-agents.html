왜 Agent만 '다시 시도'를 할 수 있나

"범용 AI 도구(ChatGPT, Gemini)는 못 하고 Agent(Claude Code, Codex)는 할 수 있다"는 차이의 진짜 이유. 모델의 능력 차이가 아닙니다. 모델을 감싼 구조의 차이입니다.

이 글은 비전공자도 따라올 수 있게 썼지만, 메인 가이드보다 한 단계 깊은 내용입니다.

01한 줄 답

모델이 더 똑똑해서가 아니라,
모델이 "또 시도하겠다"고 말하면
그걸 실제로 또 돌려주는 자동 루프가 있기 때문입니다.

— 같은 모델을 채팅에 넣어도, Agent에 넣어도, 모델은 똑같습니다. 다른 건 그 주변.

02LLM은 '상태 없는 함수'다

모든 이야기는 여기서 시작됩니다. LLM(거대언어모델 · Claude·GPT 등)의 본질은 의외로 단순합니다.

정의

LLM = "텍스트를 넣으면 텍스트가 나오는 함수" 하나.

한 번 부르면 한 번 답하고 끝. 기억도 없고, 자기가 방금 뭘 했는지도 모릅니다. 다음에 또 부르면 처음부터입니다.

이 점이 직관과 매우 다릅니다. ChatGPT랑 "어제 이야기"를 이어 하면 마치 모델이 기억하는 것 같지만, 사실 그건 대화 시스템이 어제 대화 전체를 다시 모델에 넣어주기 때문입니다. 모델 자체는 어제를 모릅니다.

비유하자면 LLM은 매번 처음 만나는 직원입니다. 머리는 비상하지만, 어제 한 일을 묻고 싶으면 어제 일기를 다시 보여줘야 합니다.

핵심: 이후 모든 이야기는 "이 상태 없는 함수에게 어떤 보조 장치를 붙이느냐"의 차이입니다.

03"메모리"라는 거짓말 — 컨텍스트 윈도우의 정체

흔히 "AI에게 메모리가 있다"고 말합니다. 정확히 말하면 없습니다. 있는 건 컨텍스트 윈도우예요.

컨텍스트 윈도우

모델을 한 번 부를 때 입력으로 한꺼번에 넣을 수 있는 텍스트의 최대 길이. 현재 Claude는 약 20만 토큰(영어 기준 책 한 권 분량), GPT-4o는 12만 토큰.

"대화 메모리"라 부르는 것의 정체:

  • 채팅창에 메시지를 쓰면 그 대화 전체 히스토리가 매번 모델에 다시 들어갑니다.
  • 대화가 길어져 윈도우 한도에 가까워지면, 시스템이 앞부분을 잘라내거나 요약해서 줄입니다.
  • 그래서 "앞에서 한 약속을 까먹는" 증상이 생기는 거예요.

"장기 기억"은 모델 안엔 없습니다. 별도로 파일·DB에 기록해두고, 필요할 때마다 컨텍스트에 다시 넣어주는 방식으로 흉내냅니다. ChatGPT의 "Memory" 기능, Claude의 프로젝트 파일, AI 노트앱들 — 다 같은 원리입니다.

실무 함의: 그래서 대화가 너무 길어지면 새 대화를 시작하고 핵심 맥락만 다시 넣어주는 게 결과가 좋습니다. 메인 가이드의 "막혔을 때 — 증상 1"이 이 이유예요.

04시스템 프롬프트 — 매 호출 앞에 붙는 명령

모델은 매 호출마다 "너는 누구이고 어떻게 행동해야 하는지"를 다시 듣습니다. 이게 시스템 프롬프트입니다.

시스템 프롬프트

사용자 메시지 앞에 매번 자동으로 붙는 메타 명령. 사용자는 보통 못 봅니다. 도구 만든 회사가 정해둡니다.

같은 모델이라도 시스템 프롬프트가 다르면 완전히 다른 도구가 됩니다.

도구시스템 프롬프트 (대략 이런 톤)
ChatGPT 웹 "너는 친절한 AI 어시스턴트다. 한 번에 답하고, 사용자 다음 메시지를 기다려라."
Claude Code "너는 사용자의 PC에서 일하는 코딩 동료다. 파일을 읽고 쓸 수 있고 검색·실행 도구가 있다. 임무를 받으면 도구를 골라 써서 끝까지 진행하라. 위험한 작업은 사용자에게 먼저 확인해라."

핵심 차이: "끝까지 진행하라"가 들어가 있느냐. 이 한 줄이 모델의 행동을 바꿉니다. 한 번 답하고 멈추는 대신, 도구를 부르고 결과를 보고 다음 단계를 결정합니다.

실무 함의: 여러분이 채팅창에 "단계별로 진행해줘"라고 쓰는 건 사실 시스템 프롬프트에 임시로 한 줄을 더하는 것과 비슷한 효과입니다. 그래서 효과가 있어요.

05도구(Tools) — 모델이 "이걸 부르고 싶다"고 말하는 법

모델은 PC 안의 파일을 직접 못 읽습니다. 인터넷 검색도 못 합니다. 모델이 할 수 있는 건 오직 텍스트 출력뿐이에요. 그런데 어떻게 파일을 읽나요?

방법은 약속된 형식으로 "이 도구를 부르고 싶다"고 텍스트로 말하는 겁니다. 예를 들어 모델이 이렇게 출력했다고 합시다:

모델의 출력
read_file("시험자료/A중_2024_중간_국어.pdf")

모델은 이걸 그냥 텍스트로 쓴 거예요. 모델은 파일을 읽지 않았습니다. "읽고 싶다"고 말했을 뿐입니다. 그 다음에 일어나는 일:

  1. 모델 응답을 받은 주변 프로그램(=하니스)이 이 텍스트를 해석합니다: "아, read_file이라는 도구를 호출하라고 했네."
  2. 하니스가 실제로 그 PDF 파일을 엽니다 (이건 그냥 평범한 컴퓨터 동작).
  3. 파일 내용을 텍스트로 만들어 다음 번 모델 호출의 컨텍스트에 넣어줍니다.
  4. 모델은 그 내용을 보고 다음 행동을 결정합니다.
도구 (Tool)

모델이 "사용하고 싶다"고 텍스트로 표현할 수 있는 외부 함수. 실제 실행은 모델이 하는 게 아니라 주변 프로그램이 합니다.

예시: 파일 읽기·쓰기, 폴더 목록 보기, 웹 검색, 코드 실행, 데이터베이스 조회, 다른 사이트 API 호출 등.

범용 채팅 도구도 일부 도구를 가지긴 합니다 (ChatGPT의 "코드 인터프리터", 이미지 생성 등). 하지만 Agent는 도구 종류가 훨씬 많고, 도구를 연달아 여러 번 부르는 자유도가 큽니다.

06하니스(Harness) — Agent의 본질

위에서 슬쩍 언급한 "주변 프로그램"이 진짜 주인공입니다.

하니스 (Harness)

모델을 부르고 → 응답을 받고 → 도구 호출이 있으면 실행하고 → 결과를 다시 모델에 먹이고 → 이걸 모델이 "끝났다"고 할 때까지 자동으로 반복하는 프로그램.

한국어로 옮기면 "마구" "받침대" — 모델이라는 동물에게 씌우는 보조 장치라는 뉘앙스입니다.

하니스가 하는 일을 코드 흉내로 적으면:

하니스가 도는 루프 (의사 코드)
context = [시스템 프롬프트, 사용자 임무] 반복: 응답 = 모델_호출(context) context.추가(응답) 만약 응답.도구_호출_있음: 결과 = 도구_실행(응답.도구_호출) context.추가(결과) 아니면: # 모델이 "완료"라고 답함 반복_종료 사용자에게 보고

이 작은 루프가 Agent의 거의 전부입니다. 이 루프가 없으면 같은 모델도 한 번에 답하고 멈춥니다. 루프가 있으면 자기 결과를 보고 다음 행동을 정합니다.

비유: 모델은 똑같이 똑똑한 직원. 범용 채팅은 "한 마디 시키고 답 들으면 끝나는 면접". 하니스는 "임무 주고 사무실에 풀어둔 상태 — 본인이 스스로 굴러감".

핵심 한 마디: 가운데 LLM만 보면 범용 AI도 Agent도 똑같습니다. 주변 점선 박스(하니스)와 양옆 도구 박스가 있느냐 없느냐가 모든 차이를 만듭니다.

각 구성요소를 한 문장으로

  • LLM — 텍스트를 받아 텍스트를 만드는 함수. 똑똑하지만 손발도 기억도 없음.
  • 시스템 프롬프트 — 모델에게 "너는 누구냐, 어떻게 행동할 거냐"를 매번 다시 알려주는 한두 문단의 명령.
  • 컨텍스트 윈도우 — 모델이 한 번에 볼 수 있는 입력 전체. 대화·도구결과·임무가 다 여기 쌓임. 유한.
  • 도구 — 모델이 "쓰고 싶다"고 말할 수 있는 외부 기능. 실제 실행은 하니스가 함.
  • 하니스 — 위 네 가지를 묶어서 자동으로 돌리는 프로그램. Agent를 Agent로 만드는 핵심.

08두 개의 다른 루프

범용 AI와 Agent의 흐름을 나란히 놓으면 차이가 또렷합니다.

① 범용 AI의 루프 — 루프 도는 주체: 사람

범용 채팅 (ChatGPT 웹, Claude 웹 등)
사용자 메시지 → 모델 한 번 호출 → 응답 표시 ↑ 사용자가 평가하고 직접 다음 메시지를 입력 ↻ (이 루프를 도는 건 사람)

② Agent의 루프 — 루프 도는 주체: 프로그램(하니스)

Agent (Claude Code, Codex 등)
사용자 임무 → 모델 호출 ↓ 모델 응답: "도구 X 부르겠다" ↓ 도구 실제 실행 (파일 열기, 검색 등) ↓ 결과를 컨텍스트에 추가 → 모델 다시 호출 ↻ 모델이 "완료"라고 할 때까지 자동 반복 ↓ 사용자에게 결과 보고

같은 모델, 같은 컨텍스트 윈도우, 같은 도구 호출 능력 — 차이는 한 가지: 루프를 누가 도느냐.

09그래서 "다시 시도"가 가능한 진짜 이유

위 그림이 머릿속에 그려졌다면, 이제 본 질문에 답할 수 있습니다.

"부족하면 다시"가 Agent에서 가능한 이유는 이렇게 분해됩니다:

  1. 모델이 도구를 부른다 (예: 웹 검색).
  2. 도구가 결과를 돌려준다 (예: "검색 결과 0건").
  3. 그 결과가 다음 모델 호출의 컨텍스트에 자동으로 추가된다.
  4. 모델은 그 결과를 보고 "음, 키워드가 안 좋았네. 다른 단어로 다시 검색해야겠다"고 결정한다.
  5. 모델이 또 다른 도구 호출을 출력 → 하니스가 또 실행 → ...
  6. 모델이 만족할 때까지 이 루프가 자동으로 돈다.

중요한 건 이 모든 게 "한 번의 사용자 임무" 안에서 자동으로 일어난다는 점입니다. 사용자는 가만히 있어도 됩니다.

범용 채팅에서 똑같은 효과를 내려면:

  • 사용자가 결과를 보고 → "이 키워드 결과 0건이네, 다른 단어로 다시 검색해줘" → 모델이 답 → ...
  • 매 단계마다 사람이 끼어들어야 함.
정리: "다시 시도" 능력은 모델의 능력이 아니라 루프의 자동화 수준의 문제입니다. 모델 자체는 이미 "다시 해야겠다"고 판단할 줄 압니다. 다만 그 판단을 받아 실제 실행으로 옮겨주는 손발(=하니스)이 범용 채팅에는 없을 뿐.

자기 검증(self-check)도 같은 원리

"네가 만든 답이 맞는지 다시 확인해줘"라고 시키면, Agent는 ① 자기가 방금 만든 파일을 도구로 다시 읽어 → ② 검토 → ③ 틀리면 수정 → ④ 다시 검토 — 이 사이클을 자동으로 돕니다. 같은 모델을 범용 채팅에 두면 ①에서 막힙니다 ("저는 파일을 읽을 수 없습니다").

10한계는 어디에 있나

Agent가 만능은 아닙니다. 구조적 한계도 같은 자리에서 나옵니다.

한계 1 — 컨텍스트 윈도우는 유한하다

루프가 길어지면 도구 호출 결과들이 컨텍스트에 계속 쌓입니다. 20만 토큰을 넘기면 앞부분이 잘리거나 요약됩니다. 그래서 한 임무가 너무 길면 "앞에서 한 약속을 깬다"가 일어나요.

대응: 큰 일을 작은 임무로 쪼개기. 각 임무의 결과를 파일로 저장하고, 다음 임무는 그 파일을 다시 읽어 시작.

한계 2 — 잘못된 판단도 자동으로 계속 한다

모델이 한 번 잘못된 방향을 잡으면, 루프가 그걸 자동으로 굴려서 멀리 갑니다. 30분 후에 보니 산으로 가 있는 케이스.

대응: 메인 가이드 Step 3 — "바로 실행 말고 계획만 먼저". 인간이 중간에 끼어드는 체크포인트를 명시적으로 만들기.

한계 3 — 도구의 한계가 곧 Agent의 한계

Agent가 PDF에 그래프가 있는데 그래프 데이터를 읽을 수 있는 도구가 없으면 못 합니다. 모델이 똑똑해도 손이 없으면 못 짚는 거예요.

대응: 어떤 도구가 있는지 알고 시작. 없는 일은 사람이 우회 (그래프 부분만 따로 캡쳐해서 별도 처리 등).

한계 4 — 권한 = 위험

Agent가 파일을 쓰고 지울 수 있다는 건, 잘못된 판단으로 중요한 파일을 덮어쓸 수도 있다는 뜻입니다.

대응: 작업 폴더를 분리. 원본 파일은 별도 폴더에 보관. Agent에게는 작업 사본만 노출. Git 같은 버전관리도 안전망.

핵심: 자동화의 힘은 곧 자동화된 실수의 힘입니다. "끝까지 자동으로"는 매력적이지만, 그 자동의 출발 지점(계획)과 도착 지점(검수)에 사람이 있어야 안전합니다.
